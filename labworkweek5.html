Week 5 – Standards in Curation

Designing a metadata schema and embedding it into my encoded Zelda score.

Scroll Down to Explore Week 5 IL Tasks

Metadata Schema and MEI Header

This week’s activity focused on designing a reusable metadata schema and embedding it into my MEI file so that each encoded piece in my future catalogue can be described in a consistent, machine-readable way. Using my ongoing theme of Koji Kondo’s music for The Legend of Zelda, I enriched the MEI header of my Great Fairy Fountain transcription with structured metadata and links to external authorities.

To complete this activity, I created a schema with six metadata elements, added them into the of my Week 3 MEI file, and validated the result using MEI Friend.

1. Metadata Schema Design

For this portfolio, I defined a simple yet extensible metadata schema to be applied to every encoded piece in my catalogue. It includes the required fields—title, artist, and publisher—as well as additional elements that reflect the needs of my specific project.

Schema Elements:

  • Title – the main title of the piece as represented in the score.
  • Composer (Artist) – the original creator of the music (Koji Kondo).
  • Arranger – identifies the creator of the piano etude arrangement.
  • Publisher – the company responsible for the original game and music (Nintendo).
  • Encoding Date – the date on which this digital MEI version was created.
  • Source Identifier (URI) – a link to the original digital source (MuseScore score page).
  • Genre, Key, Meter – additional musical descriptors used in the work description.

Together, these elements support both bibliographic description (who, when, publisher, source) and musical description (genre, key, meter), creating a catalogue useful to users such as musicologists and performers.

2. Enriching the MEI Header

I then modified the <meiHead> of my Week 3 MEI file to reflect this schema. The example below shows the core metadata encoded in the header, including the use of @auth.uri to link the composer to a stable external authority entry.

MEI Header Excerpt:

<meiHead>
  <fileDesc>
    <titleStmt>
      <title>Great Fairy Fountain</title>

      <respStmt>
        <persName role="composer"
                  auth.uri="https://www.wikidata.org/wiki/Q51308">
          Koji Kondo
        </persName>
        <persName role="arranger">Erik Correll</persName>
        <persName role="encoder">2742018</persName>
      </respStmt>
    </titleStmt>

    <pubStmt>
      <publisher>Nintendo</publisher>
      <date isodate="2025-11-14">2025</date>
      <availability>All Rights Reserved</availability>
    </pubStmt>

    <sourceDesc>
      <source>
        <identifier type="URI">
          https://musescore.com/user/81591427/scores/27189142
        </identifier>
        <titleStmt>
          <title>Great Fairy Fountain</title>
          <title type="uniform">Piano Etude Arrangement</title>
        </titleStmt>
      </source>
    </sourceDesc>
  </fileDesc>

  <workDesc>
    <genre>Game Music / Ambient Theme</genre>
    <key>A minor</key>
    <meter>3/4</meter>
  </workDesc>
</meiHead>

The @auth.uri attribute on the composer’s <persName> element links to Koji Kondo’s Wikidata entry, which provides a stable authority reference that can be reused across different catalogues and projects. The <sourceDesc> section records the transcription's origin, enabling future users to trace the edition and context of this digital score.

3. Validation with MEI Friend

To ensure that my MEI header complied with the official MEI metadata standards, I edited and validated the file using MEI Friend. These tools highlight structural and attribute errors (for example, misplaced elements or unsupported attributes) with red underlining, making it easier to correct mistakes.

After several iterations adjusting the placement of <fileDesc>, <pubStmt>, <sourceDesc>, and <workDesc>, I was able to create a header structure that validates cleanly against the MEI schema, with no remaining error markers.

4. Reflection

This task highlighted the importance of well-structured metadata in digital music archives. Even though the musical content of the piece did not change, enriching the MEI header with a clear schema, authority-linked names, and source information fundamentally improved the file's reusability.

By designing a schema that includes title, composer, arranger, publisher, date, and source URI, and by encoding additional musical descriptors such as genre, key, and meter, I have created a template that can be reused across other Zelda-related scores. This will be especially useful in later weeks when I begin to use this metadata on the web and combine it with other analytical outputs.
